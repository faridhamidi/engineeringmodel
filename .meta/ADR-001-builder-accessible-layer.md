# ADR-001: Deliver The Engine Through Native Steering And A Skill

## Decision

The builder-accessible layer translates the existing methodology without becoming a
second methodology.

- Human-facing consequence questions live in `builders/` and link to the canonical
  engine in `core/` and `governed-automation/`.
- Always-on steering uses a marked, fail-closed block installed in native instruction
  surfaces. `AGENTS.md` and `CLAUDE.md` are supported and parity-checked forms.
- On-demand depth uses the model-invoked, Agent Skills-compatible
  [`engineering-model` skill](../skills/engineering-model/SKILL.md).
- Standalone skill references are generated byte-for-byte from canonical engine files.
  Generated copies are delivery projections, not independent sources of truth.
- Git supplies local authoring checkpoints. It does not reverse external effects.
- Every external-substrate effect requires explicit human approval for its exact
  target, consequence, and action. Steering is procedural; hard enforcement belongs in
  the affected substrate.
- The builder layer uses direct structural and oracle witnesses, not a manifest-backed
  conformance harness.

## Rationale

An on-demand skill cannot make a rule continuously present, while native steering
cannot carry the full methodology without imposing excessive context load. Combining a
small always-on block with progressively disclosed skill references gives each concern
one appropriate surface.

Standalone installation requires the skill to carry its references. Mechanical parity
preserves canonical ownership while avoiding broken paths after installation.

## Consequences

- Installing the skill alone is incomplete; every active runtime also needs its native
  steering surface installed.
- A separately installed skill is a point-in-time copy and must be upgraded to receive
  later engine changes.
- Structural checks prove package shape, reference parity, and decision-oracle behavior.
  They do not prove agent obedience, client installation behavior, substrate controls,
  or operational risk reduction.
- New native runtimes require a placement template and parity evidence before being
  claimed as supported.

## Evidence

**Implemented, tested:** builder surfaces, `AGENTS.md` and `CLAUDE.md` steering,
installable skill packaging, canonical-reference synchronization, minimal falsifiers,
and CI execution.

**Not demonstrated:** runtime adherence across every agent, automatic client
installation and upgrade, substrate enforcement, and operational outcomes.
